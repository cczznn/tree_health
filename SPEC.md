# SPEC.md

## 1. 问题陈述

### 1.1 背景
很多普通用户在日常饮食、运动和体重管理中面临三个持续性问题：

1. 不知道每天应该吃多少、吃什么更合适
2. 很难持续、清晰地记录饮食和运动行为
3. 缺少一个轻量、可操作、可持续使用的工具来跟踪健康变化

### 1.2 目标
本项目要实现一个面向大众用户的健康管理微信小程序应用，帮助用户：

- 快速记录饮食
- 查看热量与营养素摄入情况
- 搜索并添加食物到记录中
- 生成基础 AI 推荐的饮食/健身建议
- 记录体重与围度变化
- 生成和打卡基础健身计划

### 1.3 产品与实现约束
- 本项目最终交付形态为微信小程序，上线目标也是微信小程序
- 前端界面必须体现“小程序布局感”，包含底部 tab 栏与简洁卡片式信息结构
- 架构采用单仓库前后端分层实现
- AI 推荐采用“结构化规则 + LLM 文本生成”的混合方式，规则优先，LLM 负责组织更自然、可读、可执行的建议文本
- 需要登录与用户隔离，所有核心数据按用户维度保存
- 单容器优先，只有在确实拆分为独立前后端服务时才引入 `docker-compose.yml`

### 1.3 为什么值得做
这是一个真实且常见的生活问题，用户几乎每天都会遇到。相比复杂的专业健康平台，本项目强调：

- 轻量
- 易用
- 低门槛
- 可持续记录
- 有明确反馈

适合做成一个中等规模、模块清晰、便于测试和迭代的课程项目。

### 1.4 目标用户
- 大众通用用户
- 有饮食记录习惯或意愿的人
- 想更健康地吃、规律运动、观察身体变化的人
- 希望在吃饭前快速参考食物营养信息的人
- 需要登录后保存个人饮食、运动和身体数据的单用户用户群体

## 2. 用户故事

以下用户故事遵循 INVEST 原则，覆盖核心使用场景。

### US1：记录一餐饮食
作为一名用户，我希望能快速搜索并添加食物到某一餐中，这样我可以在几秒钟内完成饮食记录。

**验收标准**
- 用户可以通过关键词搜索食物
- 用户可以从预置食物库中选择食物
- 用户可以手动添加自定义食物
- 用户可以为食物填写数量或份量

### US2：查看每日热量与营养摄入
作为一名用户，我希望看到当天累计摄入的热量和主要营养素，这样我能知道自己的饮食是否合理。

**验收标准**
- 系统展示每日总热量
- 系统展示蛋白质、脂肪、碳水、膳食纤维等核心营养素
- 数据会随着新增饮食记录实时更新

### US3：获取饮食与运动建议
作为一名用户，我希望系统根据我的饮食记录和身体数据给出基础 AI 推荐建议，这样我能更容易做出健康选择。

**验收标准**
- 系统可根据用户当天摄入与目标给出建议
- 建议内容包含饮食调整或运动建议
- 建议具备基础可解释性，不只是输出结论

### US4：搜索食物营养信息
作为一名用户，我希望在吃饭前搜索某种食物的营养信息，这样我能快速判断是否适合当前目标。

**验收标准**
- 用户可以按名称搜索食物
- 搜索结果包含热量和主要营养素
- 用户可以查看食物详情页

### US5：制定并打卡健身计划
作为一名用户，我希望生成一个简单的健身计划并进行打卡，这样我能更规律地运动。

**验收标准**
- 用户可以生成基础健身计划
- 用户可以查看计划中的训练安排
- 用户可以对训练项完成打卡

### US6：记录体重与围度变化
作为一名用户，我希望记录体重和围度等身体数据，这样我能观察一段时间内的变化趋势。

**验收标准**
- 用户可以新增体重记录
- 用户可以新增围度记录
- 系统可以查看历史趋势

### US7：管理自定义食物
作为一名用户，我希望能把常吃但库里没有的食物手动添加进去，这样我能更完整地记录饮食。

**验收标准**
- 用户可以创建自定义食物
- 自定义食物支持填写热量和主要营养素
- 自定义食物会出现在搜索和记录中

## 3. 功能规约

### 3.1 模块一：饮食记录

#### 输入
- 食物名称
- 食物份量/数量
- 餐次（早餐、午餐、晚餐、加餐）
- 时间
- 备注（可选）

#### 行为
- 用户从食物库选择食物或自定义添加食物
- 系统计算该条记录对应的营养摄入
- 用户可查看当天饮食列表
- 用户可编辑或删除记录

#### 输出
- 单条饮食记录
- 当日饮食汇总
- 热量与营养素统计

#### 边界条件
- 份量为空时不可提交
- 食物不存在时允许自定义创建
- 同一食物可多次添加到不同餐次

#### 错误处理
- 搜索无结果时提示用户创建新食物
- 输入格式错误时阻止提交并提示修正

---

### 3.2 模块二：热量统计与营养素统计

#### 输入
- 当日饮食记录
- 用户身体目标（减脂、维持、增肌）
- 计划摄入目标（可选）

#### 行为
- 汇总每日总热量
- 汇总核心营养素：蛋白质、脂肪、碳水、纤维、糖、钠等
- 对比目标值给出简单提醒

#### 输出
- 每日热量统计卡片
- 营养素分布视图
- 目标完成度提示

#### 边界条件
- 当天无记录时显示空状态
- 目标缺失时仅展示统计，不做目标对比

#### 错误处理
- 数据缺失时按可用字段计算并提示部分统计

---

### 3.3 模块三：食物库与搜索

#### 输入
- 关键词
- 分类筛选（可选）
- 自定义食物信息

#### 行为
- 提供预置食物库
- 支持按关键字搜索
- 支持查看食物详情
- 支持新增自定义食物并加入食物库
- 自定义食物与预置食物共享同一搜索接口

#### 输出
- 搜索结果列表
- 食物详情页
- 自定义食物创建结果

#### 边界条件
- 搜索关键词为空时展示最近使用或热门食物
- 重名食物允许存在多个来源，但数据层需要保留来源字段以便区分
- 搜索结果默认展示保持简洁，不强制显示来源标签

#### 错误处理
- 搜索失败时显示可重试状态
- 自定义食物保存失败时保留输入内容

---

### 3.4 模块四：AI 推荐

#### 输入
- 用户饮食记录
- 用户健身目标
- 用户体重/围度数据
- 当日或周期性统计结果

#### 行为
- 采用“结构化规则 + LLM 文本生成”的混合方式生成建议
- 规则负责判断摄入状态、训练状态和身体变化
- LLM 负责把建议组织成更自然、可读、可执行的文本
- 基于目标和当前摄入生成简单运动建议
- 输出尽量明确、可执行、可解释

#### 输出
- 饮食建议
- 运动建议
- 今日优化建议摘要

#### 边界条件
- 当数据不足时，AI 推荐应降级为通用建议
- 不允许输出高风险医疗结论
- 不做开放式聊天机器人

#### 错误处理
- 推荐生成失败时返回默认建议模板
- 当输入过少时提示用户先补充记录

> 注：这里的 AI 推荐应优先采用“受约束的生成 + 规则兜底”，避免变成不可控的聊天机器人。

---

### 3.5 模块五：健身计划与打卡

#### 输入
- 目标（减脂/增肌/维持）
- 训练频率
- 单次时长
- 训练偏好（可选）

#### 行为
- 系统生成基础训练计划模板
- 用户查看每周训练安排
- 用户对完成的训练进行打卡

#### 输出
- 训练计划列表
- 打卡状态
- 完成率统计

#### 边界条件
- 用户信息不足时使用默认计划模板
- 同一训练项可重复打卡，但需要按日期区分

#### 错误处理
- 计划生成失败时展示默认新手计划

---

### 3.6 模块六：体重 / 围度记录

#### 输入
- 体重
- 腰围、臀围、大腿围等围度（可选）
- 日期
- 备注（可选）

#### 行为
- 用户可添加身体数据记录
- 系统展示趋势变化
- 用于辅助 AI 推荐与目标判断

#### 输出
- 历史记录列表
- 趋势视图
- 当前变化总结

#### 边界条件
- 数据单位需统一
- 同一天重复记录需要支持覆盖或新增策略
- 趋势视图第一版采用简单列表 + 最近变化值，不做复杂图表

#### 错误处理
- 非法数值不允许保存

## 4. 非功能性需求

### 4.1 性能
- 常用页面应在可接受时间内完成首屏加载
- 搜索结果应在低延迟下返回
- 本地统计计算应尽量在前端快速完成

### 4.2 可用性
- 页面结构应尽量符合微信小程序的常见使用习惯
- 核心操作应尽量少步骤完成
- 空状态、无结果、错误状态需要清晰提示

### 4.3 可观测性
- 关键操作应有日志或可追踪记录
- 记录创建、修改、删除应便于调试

### 4.4 安全性
- 不保存敏感医疗数据
- AI 推荐需避免高风险医疗建议
- 用户自定义输入需做基础校验

### 4.5 可维护性
- 食物库、计划模板、建议规则应模块化管理
- 业务逻辑与 UI 分离

## 5. 系统架构

### 5.1 架构概览
项目采用单仓库前后端分层架构，推荐如下结构：

- **前端**：基于 Taro 的小程序端界面，最终编译对齐微信小程序
- **后端 API**：提供食物库、记录、计划、建议等接口
- **领域层**：封装业务规则、校验、统计与推荐逻辑
- **数据存储**：保存用户记录、食物库、自定义食物、计划模板
- **AI 服务层**：根据结构化数据生成建议

### 5.2 数据流
1. 用户在前端录入饮食、运动或身体数据
2. 前端提交到 API
3. 后端写入数据库并更新统计
4. 规则层先计算结构化结果，再交给 LLM 生成可读建议
5. 前端展示统计与建议结果

### 5.3 外部依赖
- AI 模型 API（如使用）
- 数据库
- 前端 UI 组件库（如适用）

- date

### 6.2 关系说明
- User 关联多个 MealRecord
- User 关联多个 WorkoutPlan
- User 关联多个 WorkoutCheckin
- User 关联多个 BodyMetric
- User 关联多个 Recommendation
- Food 可被多个 MealRecord 引用

### 6.3 约束
- Food.name 需可搜索
- MealRecord.amount 不能为负
- BodyMetric 数值字段不能为负
- 同一用户同一天的数据要可查询
- 自定义食物仅归属创建它的用户，不与其他用户共享
- 自定义食物数量上限为每用户 50 个，避免无限增长

## 7. API 设计

> 注：以下为推荐接口草案，具体实现时可根据前后端结构调整。

### 7.1 食物库
- `GET /foods?query=`：搜索食物（预置食物与自定义食物共享同一接口）
- `GET /foods/:id`：查看食物详情
- `POST /foods`：创建自定义食物
- `GET /foods/mine`：查看当前用户的自定义食物（如需要）

### 7.2 饮食记录
- `GET /meal-records?date=`：获取某日记录
- `POST /meal-records`：新增饮食记录
- `PUT /meal-records/:id`：编辑饮食记录
- `DELETE /meal-records/:id`：删除饮食记录

### 7.3 统计
- `GET /stats/daily?date=`：获取每日热量与营养统计
- `GET /stats/trend?range=`：获取趋势统计

### 7.4 健身计划
- `POST /workout-plans/generate`：生成基础计划
- `GET /workout-plans/:id`：查看计划详情
- `POST /workout-checkins`：提交打卡

### 7.5 身体数据
- `GET /body-metrics?range=`：获取身体数据趋势
- `POST /body-metrics`：新增体重/围度记录

### 7.6 AI 推荐
- `POST /recommendations/generate`：根据当前数据生成建议
- `GET /recommendations?date=`：查看历史建议

### 7.7 常见错误码
- `400`：参数错误
- `404`：资源不存在
- `409`：重复资源或冲突
- `500`：服务内部错误

## 8. 技术选型与理由

### 8.1 前端
- **建议**：Taro + TypeScript + React
- **理由**：既能保持组件化与 React 风格的开发体验，又能编译生成微信小程序代码，满足“最终上线到微信小程序”的目标

### 8.2 后端
- **建议**：Node.js + TypeScript
- **理由**：与前端同语言栈，适合快速开发 API 与 AI 推荐逻辑

### 8.3 数据库
- **建议**：MySQL
- **理由**：当前冷启动验证已确认采用 MySQL，便于后续保持与线上部署一致的数据库行为；如需本地轻量开发，可在不改变数据模型的前提下切换到 SQLite 作为替代方案

### 8.4 AI 推荐实现
- **建议**：结构化规则 + LLM 生成结合
- **理由**：既能满足“AI 推荐”要求，也能通过规则降低不稳定性

### 8.5 设计风格
- **风格**：微信小程序风格
- **理由**：符合目标用户预期，界面轻量、熟悉、易上手

## 10. 风险与未决问题

### 10.1 风险
- AI 推荐可能出现不稳定输出，需要规则兜底
- 食物营养数据来源可能不完整
- 课程时间有限，功能范围容易膨胀
- 数据模型如果设计过重，会拖慢实现进度

### 10.2 未决问题
1. AI 推荐具体采用哪种模型提供商与调用方式
2. 食物库的初始数据量要多大
3. 自定义食物数量上限设为多少更合理
4. 微信小程序与 Web 预览版是否保持完全一致的视觉实现
5. 前后端分层在代码仓库中的目录边界如何最终切分

## 11. MVP 范围建议

为了保证项目可交付，第一版建议优先实现：

1. 饮食记录
2. 食物搜索 / 食物库 / 自定义添加
3. 热量与营养素统计
4. 健身计划生成
5. 运动打卡
6. 体重 / 围度记录
7. 基础 AI 推荐

其中最先落地的核心链路是：

> 搜索食物 → 添加饮食记录 → 自动统计热量与营养素 → 生成建议

## 12. 结论

本项目是一个面向大众用户的轻量健康管理工具，重点在于把饮食、运动和身体变化记录统一到一个简单、可持续使用的小程序风格应用中。它具备真实问题、明确用户、可拆分模块和适合 TDD / subagent 开发的结构，非常适合作为 AI4SE 课程期末项目。
