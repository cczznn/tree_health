# SPEC_PROCESS.md

> 记录本项目在 Superpowers `brainstorming` 与 `writing-plans` 阶段中，对需求、边界和实现路径的澄清与修订过程。

---

## 1. 过程概述

本项目最初的方向是一个“健康 / 健身 / 饮食类小程序”，随后在多轮 brainstorming 中逐步收敛为一个具备明确边界的健康管理微信小程序风格应用。整个过程的核心不是直接决定功能清单，而是先把以下问题澄清清楚：

- 目标用户到底是谁
- 用户最常遇到的核心问题是什么
- 哪些功能属于 MVP，哪些属于扩展
- AI 推荐到底采用什么实现方式
- 项目最终要做到什么产品形态与技术边界

在这个过程中，最重要的变化是：

1. 由“泛健康工具”收敛为“面向大众通用用户的健康管理小程序风格应用”
2. 由“是否做 AI 推荐不明确”收敛为“结构化规则 + LLM 文本生成”的混合方式
3. 由“是否做成 Web 小程序风格”收敛为“最终上线到微信小程序，并具备真实布局感”
4. 由“食物搜索与食物数据来源不明确”收敛为“预置食物与自定义食物共用同一搜索接口，但数据层保留来源字段”
5. 由“是否多用户”收敛为“需要登录与用户隔离”

## 2. brainstorming 关键节点

### 2.1 关键迭代一：目标用户与核心痛点

#### 对话节选
- AI 追问：你的主要用户是学生、上班族、健身初学者，还是更细分的人群？
- 我回应：先做大众通用用户，覆盖每天记录饮食、健身后打卡、吃饭时快速参考这三类高频场景。
- AI 继续追问：你最想解决的一个核心痛点是什么，是不知道吃多少、怎么搭配，还是怎么安排运动？
- 我确认：核心问题是三件事——不知道每天吃多少合适、不知道怎么搭配饮食、也不知道怎么安排健身计划。

#### 处理决策
我采纳了 AI 的追问方式，并把原本过于宽泛的“健康/健身/饮食类小程序”收敛成“面向大众通用用户的综合型健康管理工具”。

#### 起始状态
最初的描述只有“做一个健康、健身、饮食类的小程序”，范围很大，容易变成：
- 纯饮食记录
- 纯健身打卡
- 纯 AI 问答
- 或者一个没有明确主线的综合工具

#### 关键问题
围绕以下维度进行澄清：
- 目标用户是谁
- 他们在什么场景下使用
- 最想解决的核心问题是什么
- 这个工具是记录型、建议型、计划型还是综合型

#### 结果
最终确认：
- 目标用户是**大众通用用户**
- 高频场景是**每天记录饮食、健身后打卡、吃饭时快速参考**
- 核心问题是：
  1. 不知道每天吃多少合适
  2. 不知道怎么搭配饮食
  3. 不知道怎么安排健身计划
- 工具定位为**综合型健康管理工具**

#### 影响
这一轮之后，项目不再被理解成“单一功能工具”，而是一个围绕健康记录、统计、计划和建议的闭环产品。

---

### 2.2 关键迭代二：功能范围与 MVP 裁剪

#### 对话节选
- AI 建议：先把功能分成记录、统计、计划、推荐四个大块，再决定哪些先做。
- 我追问：哪些功能必须做，哪些功能明确不做？
- AI 列举了候选功能：饮食记录、热量统计、食物搜索 / 食物库、健身计划生成、运动打卡、体重 / 围度记录、AI 推荐、社交分享、排名 / 社区、付费功能。
- 我最终决定：保留前七项，不做社交分享、付费功能、排名 / 社区。

#### 处理决策
我采纳了“先分块再裁剪”的建议，但推翻了把项目扩展成社区型产品的方向，因为那会让范围失控，也不符合课程期末项目的交付节奏。

#### 起始状态
候选功能一度包含：
- 饮食记录
- 热量统计
- 食物搜索 / 食物库
- 健身计划生成
- 运动打卡
- 体重 / 围度记录
- AI 推荐
- 社交分享
- 排名 / 社区
- 付费功能

#### 关键问题
围绕“哪些功能必须做、哪些功能明确不做”进行裁剪。

#### 结果
明确保留：
- 饮食记录
- 热量统计
- 食物搜索 / 食物库
- 健身计划生成
- 运动打卡
- 体重 / 围度记录
- AI 推荐

明确不做：
- 社交分享
- 付费功能
- 排名 / 社区

#### MVP 选择
最小版本优先落地的三个核心功能确定为：
1. 饮食记录
2. 热量统计
3. 健身计划

随后又根据课程要求和产品完整性继续扩展到完整的健康闭环。

#### 影响
这一步确保项目不会膨胀成“什么都有但什么都做不深”的大杂烩。

### 2.3 关键迭代三：AI 推荐、产品形态与技术边界

#### 对话节选
- AI 提示：推荐模块如果直接做开放式聊天，很容易变成不可控的健康问答机器人。
- 我修正：改成“结构化规则 + LLM 文本生成”的混合方式，规则优先，LLM 负责表达。
- AI 继续追问：产品最终是 Web 风格还是微信小程序风格？要不要底部 tab 栏？
- 我确认：最终要上线到微信小程序，界面必须真的有小程序布局感，并且需要底部 tab 栏。
- AI 再追问：食物数据如何处理预置和自定义的关系？
- 我确认：共用一个搜索接口，前端展示不强制区分来源，但数据层必须保留来源字段。

#### 处理决策
我采纳了 AI 对“不可控开放式推荐”的警告，并把推荐能力限定为规则和 LLM 的混合方案；同时我把产品形态从泛 Web 小程序类应用进一步收紧到“最终上线微信小程序”的交付目标。

#### 起始状态
一开始曾考虑不做 AI 推荐，避免复杂度过高。

#### 关键转折
在明确项目需要体现 AI 协作与方法论后，决定把 AI 推荐纳入核心能力，但必须是可控的、受约束的方式。

#### 结果
确认：
- AI 推荐要做
- 但不能做成开放式聊天机器人
- 推荐方式采用**结构化规则 + LLM 文本生成**的混合方案
- 规则优先，LLM 负责把结果组织成更自然、更可读的建议
- 数据不足时要降级为通用建议

#### 影响
这使得推荐模块既符合课程“AI4SE”的主题，也避免了不可控的自由生成风险。

---

### 2.4 关键迭代四：小程序形态、登录隔离与数据边界

#### 对话节选
- AI 追问：这个产品是多人共享还是每个用户自己的健康数据？
- 我确认：需要登录与用户隔离，所有饮食、身体数据、计划、推荐都按用户维度独立存储。
- AI 追问：自定义食物是否和预置食物共用检索？重名怎么办？
- 我确认：共用搜索接口，UI 不强制展示来源，但数据层必须区分来源，允许重名。
- AI 追问：趋势视图要做复杂图表还是轻量展示？
- 我确认：只做简单列表 + 最近变化值，不做复杂图表。
- AI 追问：容器化是否直接上多服务？
- 我确认：单容器优先，只有确实拆前后端服务时才用 compose。

#### 处理决策
我采纳了 AI 对数据边界和部署复杂度的提醒，并把“多用户、重名、来源、趋势、容器化”这些原本模糊的地方全部落成了可执行约束。

#### 起始状态
产品形态一开始只被描述为“小程序类应用”，实现技术路线不够明确。

#### 关键问题
进一步明确：
- 是否真的要做成小程序布局感
- 是否需要底部 tab 栏
- 最终目标平台是什么

#### 结果
最终确认：
- 必须真的做成**小程序布局感**
- 页面结构要符合微信小程序常见习惯
- 必须有**底部 tab 栏**
- 项目最终目标是**上线到微信小程序**

#### 影响
这直接影响了前端布局、页面结构和后续 T11 的任务拆分。

## 3. 关键设计决策与修订依据

### 3.1 AI 推荐实现方式

#### 决策
采用“结构化规则 + LLM 文本生成”的混合方式。

#### 原因
- 纯 LLM 容易不稳定
- 纯规则又缺少“AI 推荐”的项目特征
- 混合方式更适合课程项目，也更容易测试

#### 对 SPEC 的影响
- AI 推荐模块明确写出输入、行为、输出和降级策略
- 禁止开放式聊天机器人式实现

---

### 3.2 架构方式

#### 决策
采用单仓库前后端分层架构。

#### 原因
- 便于统一管理代码与测试
- 适合课程要求中的模块化推进
- 方便使用 worktree 和 subagent 协作

#### 对 SPEC 的影响
- 架构不再模糊为“前后端分离或轻后端”
- 统一为可控的单仓库分层方案

---

### 3.3 食物库与搜索

#### 决策
- 预置食物和自定义食物共享同一搜索接口
- 搜索结果默认展示中不强制区分来源
- 数据层保留 `sourceType`，支持重名与审计

#### 原因
- 用户体验需要简单直接
- 后端管理需要保留来源信息
- 重名数据在健康类应用里很常见，必须允许存在多个来源

#### 对 SPEC 的影响
- 食物数据模型增加来源字段
- 搜索 API 统一入口
- 任务计划中将搜索与自定义创建拆成两个任务

---

### 3.4 自定义食物归属与数量限制

#### 决策
- 自定义食物只存在对应用户的数据里
- 不与其他用户共享
- 需要限制自定义食物数量

#### 原因
- 避免数据污染和跨用户混淆
- 支持用户私有化管理
- 避免滥用和无上限增长

#### 对 SPEC 的影响
- 所有相关表都需要 `userId`
- API 与仓储层必须按用户隔离
- 需要在验收标准里体现多用户边界

---

### 3.5 趋势视图

#### 决策
趋势视图采用“简单列表 + 最近变化值”，不做复杂图表。

#### 原因
- 与项目规模相符
- 更容易实现和测试
- 更贴近“轻量可用”的产品定位

#### 对 SPEC 的影响
- 体重 / 围度模块不依赖高级可视化组件
- 前端任务可以先保证可读性，再考虑体验增强

---

### 3.6 登录与用户隔离

#### 决策
需要登录与用户隔离。

#### 原因
- 饮食、体重、围度、计划、推荐都属于明显的个人数据
- 课程项目需要体现真实产品数据边界

#### 对 SPEC 的影响
- User 实体成为核心实体之一
- 所有记录表都需要 `userId`
- 前后端任务必须考虑授权/数据隔离

---

### 3.7 容器化策略

#### 决策
单容器优先，只有确实拆前后端服务时才使用 `docker-compose`。

#### 原因
- 控制部署复杂度
- 避免为了形式而增加不必要的运维负担
- 更适合课程项目的交付节奏

#### 对 SPEC 的影响
- 容器化部分以单容器为基线
- `docker-compose` 仅作为扩展选项

## 4. 计划阶段的关键整理

在 `writing-plans` 阶段，需求被拆分为 13 个任务，以保证每个任务都能由一个 subagent 在一次会话内完成。

### 4.1 任务拆分原则
- 先底座，后功能
- 先数据层，后业务层
- 先纯逻辑，后 UI
- 先核心链路，后补充体验
- 所有任务都需要先写失败测试

### 4.2 任务间依赖关系的确认
- `T1` 提供基础工程脚手架
- `T2` 提供 domain 与 repository 基线
- `T3/T4/T7/T9` 可部分并行展开
- `T5/T10` 必须建立在统计和其他业务数据之上
- `T11` 依赖多个业务模块完成后做前端整合
- `T12` 做完整联调与验收补强
- `T13` 放在最后做容器化、CI 和文档收尾

### 4.3 为什么这样拆
这种拆法是为了满足 Superpowers 的子代理工作方式：
- 每个任务足够小
- 每个任务可以单独测试
- 每个任务都能在一个会话中推进到完成
- 失败时容易定位问题

## 5. AI 建议、采纳与修正对照

### 5.1 被采纳的建议

1. **先按目标用户和场景收敛，而不是先堆功能。**
   - 采纳原因：健康类产品如果先谈功能，很容易失去主线。

2. **把功能拆成记录、统计、计划、推荐几个大块。**
   - 采纳原因：这有助于后续写 `PLAN.md`，也方便 subagent 按模块执行。

3. **AI 推荐不要做成开放式聊天机器人，而是受约束的生成。**
   - 采纳原因：更稳定、可测、也更符合课程项目的风险控制要求。

4. **自定义食物与预置食物应共享同一搜索入口，但后端保留来源字段。**
   - 采纳原因：既保证体验简单，又保证数据可管理。

5. **趋势视图用简单列表 + 最近变化值即可。**
   - 采纳原因：项目规模与实现周期更匹配，能把精力集中到核心链路。

### 5.2 被修正或推翻的建议

1. **从“泛健康工具”修正为“面向大众通用用户的健康管理应用”。**
   - 修正原因：目标用户太泛会导致所有设计都变得模糊。

2. **从“是否做 AI 推荐不确定”修正为“必须做，但要受约束”。**
   - 修正原因：项目需要体现 AI4SE 的方法论，但不能牺牲可控性。

3. **从“Web 小程序风格”修正为“最终上线微信小程序”。**
   - 修正原因：课程交付目标需要更明确的产品方向。

4. **从“可能多人共享”修正为“需要登录与用户隔离”。**
   - 修正原因：健康数据属于强个人属性数据，必须隔离。

5. **从“复杂趋势图”修正为“列表 + 最近变化值”。**
   - 修正原因：避免把精力消耗在非关键可视化上。

## 6. Superpowers `brainstorming` 的优点与不足

### 6.1 做得好的地方

- **会主动追问关键约束。** 它不会只接受“做一个健康小程序”这种模糊描述，而是不断追问用户、场景、范围和目标。
- **能逼出隐藏假设。** 比如是否登录、是否多用户、AI 推荐是否开放式、食物库是否共享等，都是如果不问就很容易默认掉的假设。
- **有利于形成可执行的计划。** 它的追问方式天然适合把需求整理成后续 `PLAN.md` 的任务拆分。
- **会推动我收敛范围。** 在功能膨胀时，它能不断提醒哪些内容属于 MVP，哪些应延后。

### 6.2 让我不太满意的地方

- **有些问题偏通用，仍需要我主动收紧。** 它能问“要做什么”，但有时不会直接替我做出足够具体的产品约束，我仍然需要主动把答案收敛到可实现粒度。
- **对实现边界的落地还不够细。** 例如 AI 推荐、登录鉴权、食物来源区分这些问题，AI 能提醒方向，但最终还是要我自己把边界写死。
- **如果不主动追问，它可能会停留在高层建议。** 这意味着我不能把它当成自动完成 spec 的工具，而必须在它追问的基础上继续追问它。

### 6.3 总体评价

Superpowers 的 brainstorming 在这个项目里最大的价值，是把“模糊创意”变成“可开发的产品定义”。它并不能替我做产品判断，但它能迫使我把判断说清楚、写清楚，并把这些判断固化成后续实现可以遵循的约束。

## 7. 仍然保留的开放问题

虽然本轮已经把规格基本定清楚，但仍然有少量实现细节需要后续在执行阶段进一步确认：

1. 微信小程序最终运行形态如何与当前单仓库实现对接
2. AI 推荐调用哪一种 LLM 接口或模拟方式
3. 自定义食物数量限制的具体阈值
4. 登录鉴权的具体实现方式
5. 统计和推荐在前后端之间的最终分工

这些问题不会影响当前 SPEC 和 PLAN 的主线，但会在实现阶段影响具体实现细节。

## 8. 过程反思

这次 brainstorming 的最大价值，不在于“确定了做什么”，而在于“明确了不做什么”和“每个功能为什么存在”。

几个最关键的收获是：

- 如果不先明确目标用户，健康类产品很容易泛化成没有主线的功能拼盘
- 如果不先明确 AI 的边界，推荐模块会滑向不可控聊天机器人
- 如果不先明确是否需要登录与隔离，后续数据模型和 API 会反复返工
- 如果不先明确小程序布局感与最终上线目标，前端任务的设计会失焦

因此，这份 SPEC 的价值主要在于：它不仅列出了功能，还明确了产品形态、实现原则和边界约束，为后续的 `PLAN.md`、冷启动验证和实际开发提供了稳定基线。
