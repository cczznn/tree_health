# SPEC_PROCESS.md

> 记录本项目在 Superpowers `brainstorming`、`writing-plans` 以及冷启动验证阶段，对需求、边界和实现路径的澄清与修订过程。

---

## 1. 过程概述

本项目最初的方向是一个“健康 / 健身 / 饮食类小程序”，随后在多轮 brainstorming 中逐步收敛为一个具备明确边界的健康管理微信小程序风格应用。整个过程的核心不是直接决定功能清单，而是先把以下问题澄清清楚：

- 目标用户到底是谁
- 用户最常遇到的核心问题是什么
- 哪些功能属于 MVP，哪些属于扩展
- AI 推荐到底采用什么实现方式
- 项目最终要做到什么产品形态与技术边界

在这个过程中，最重要的变化是：

1. 由“泛健康工具”收敛为“面向大众通用用户的健康管理小程序风格应用”
2. 由“是否做 AI 推荐不明确”收敛为“结构化规则 + LLM 文本生成”的混合方式
3. 由“是否做成 Web 小程序风格”收敛为“最终上线到微信小程序，并具备真实布局感”
4. 由“食物搜索与食物数据来源不明确”收敛为“预置食物与自定义食物共用同一搜索接口，但数据层保留来源字段”
5. 由“是否多用户”收敛为“需要登录与用户隔离”

## 2. brainstorming 关键节点

### 2.1 关键迭代一：目标用户与核心痛点

#### 对话节选
- AI 追问：你的主要用户是学生、上班族、健身初学者，还是更细分的人群？
- 我回应：先做大众通用用户，覆盖每天记录饮食、健身后打卡、吃饭时快速参考这三类高频场景。
- AI 继续追问：你最想解决的一个核心痛点是什么，是不知道吃多少、怎么搭配，还是怎么安排运动？
- 我确认：核心问题是三件事——不知道每天吃多少合适、不知道怎么搭配饮食、也不知道怎么安排健身计划。

#### 处理决策
我采纳了 AI 的追问方式，并把原本过于宽泛的“健康/健身/饮食类小程序”收敛成“面向大众通用用户的综合型健康管理工具”。

#### 起始状态
最初的描述只有“做一个健康、健身、饮食类的小程序”，范围很大，容易变成：
- 纯饮食记录
- 纯健身打卡
- 纯 AI 问答
- 或者一个没有明确主线的综合工具

#### 关键问题
围绕以下维度进行澄清：
- 目标用户是谁
- 他们在什么场景下使用
- 最想解决的核心问题是什么
- 这个工具是记录型、建议型、计划型还是综合型

#### 结果
最终确认：
- 目标用户是**大众通用用户**
- 高频场景是**每天记录饮食、健身后打卡、吃饭时快速参考**
- 核心问题是：
  1. 不知道每天吃多少合适
  2. 不知道怎么搭配饮食
  3. 不知道怎么安排健身计划
- 工具定位为**综合型健康管理工具**

#### 影响
这一轮之后，项目不再被理解成“单一功能工具”，而是一个围绕健康记录、统计、计划和建议的闭环产品。

---

### 2.2 关键迭代二：功能范围与 MVP 裁剪

#### 对话节选
- AI 建议：先把功能分成记录、统计、计划、推荐四个大块，再决定哪些先做。
- 我追问：哪些功能必须做，哪些功能明确不做？
- AI 列举了候选功能：饮食记录、热量统计、食物搜索 / 食物库、健身计划生成、运动打卡、体重 / 围度记录、AI 推荐、社交分享、排名 / 社区、付费功能。
- 我最终决定：保留前七项，不做社交分享、付费功能、排名 / 社区。

#### 处理决策
我采纳了“先分块再裁剪”的建议，但推翻了把项目扩展成社区型产品的方向，因为那会让范围失控，也不符合课程期末项目的交付节奏。

#### 起始状态
候选功能一度包含：
- 饮食记录
- 热量统计
- 食物搜索 / 食物库
- 健身计划生成
- 运动打卡
- 体重 / 围度记录
- AI 推荐
- 社交分享
- 排名 / 社区
- 付费功能

#### 关键问题
围绕“哪些功能必须做、哪些功能明确不做”进行裁剪。

#### 结果
明确保留：
- 饮食记录
- 热量统计
- 食物搜索 / 食物库
- 健身计划生成
- 运动打卡
- 体重 / 围度记录
- AI 推荐

明确不做：
- 社交分享
- 付费功能
- 排名 / 社区

#### MVP 选择
最小版本优先落地的三个核心功能确定为：
1. 饮食记录
2. 热量统计
3. 健身计划

随后又根据课程要求和产品完整性继续扩展到完整的健康闭环。

#### 影响
这一步确保项目不会膨胀成“什么都有但什么都做不深”的大杂烩。

### 2.3 关键迭代三：AI 推荐、产品形态与技术边界

#### 对话节选
- AI 提示：推荐模块如果直接做开放式聊天，很容易变成不可控的健康问答机器人。
- 我修正：改成“结构化规则 + LLM 文本生成”的混合方式，规则优先，LLM 负责表达。
- AI 继续追问：产品最终是 Web 风格还是微信小程序风格？要不要底部 tab 栏？
- 我确认：最终要上线到微信小程序，界面必须真的有小程序布局感，并且需要底部 tab 栏。
- AI 再追问：食物数据如何处理预置和自定义的关系？
- 我确认：共用一个搜索接口，前端展示不强制区分来源，但数据层必须保留来源字段。

#### 处理决策
我采纳了 AI 对“不可控开放式推荐”的警告，并把推荐能力限定为规则和 LLM 的混合方案；同时我把产品形态从泛 Web 小程序类应用进一步收紧到“最终上线微信小程序”的交付目标。

#### 起始状态
一开始曾考虑不做 AI 推荐，避免复杂度过高。

#### 关键转折
在明确项目需要体现 AI 协作与方法论后，决定把 AI 推荐纳入核心能力，但必须是可控的、受约束的方式。

#### 结果
确认：
- AI 推荐要做
- 但不能做成开放式聊天机器人
- 推荐方式采用**结构化规则 + LLM 文本生成**的混合方案
- 规则优先，LLM 负责把结果组织成更自然、更可读的建议
- 数据不足时要降级为通用建议

#### 影响
这使得推荐模块既符合课程“AI4SE”的主题，也避免了不可控的自由生成风险。

---

### 2.4 关键迭代四：小程序形态、登录隔离与数据边界

#### 对话节选
- AI 追问：这个产品是多人共享还是每个用户自己的健康数据？
- 我确认：需要登录与用户隔离，所有饮食、身体数据、计划、推荐都按用户维度独立存储。
- AI 追问：自定义食物是否和预置食物共用检索？重名怎么办？
- 我确认：共用搜索接口，UI 不强制展示来源，但数据层必须区分来源，允许重名。
- AI 追问：趋势视图要做复杂图表还是轻量展示？
- 我确认：只做简单列表 + 最近变化值，不做复杂图表。
- AI 追问：容器化是否直接上多服务？
- 我确认：单容器优先，只有确实拆前后端服务时才用 compose。

#### 处理决策
我采纳了 AI 对数据边界和部署复杂度的提醒，并把“多用户、重名、来源、趋势、容器化”这些原本模糊的地方全部落成了可执行约束。

#### 起始状态
产品形态一开始只被描述为“小程序类应用”，实现技术路线不够明确。

#### 关键问题
进一步明确：
- 是否真的要做成小程序布局感
- 是否需要底部 tab 栏
- 最终目标平台是什么

#### 结果
最终确认：
- 必须真的做成**小程序布局感**
- 页面结构要符合微信小程序常见习惯
- 必须有**底部 tab 栏**
- 项目最终目标是**上线到微信小程序**

#### 影响
这直接影响了前端布局、页面结构和后续 T11 的任务拆分。

## 3. 关键设计决策与修订依据

### 3.1 AI 推荐实现方式

#### 决策
采用“结构化规则 + LLM 文本生成”的混合方式。

#### 原因
- 纯 LLM 容易不稳定
- 纯规则又缺少“AI 推荐”的项目特征
- 混合方式更适合课程项目，也更容易测试

#### 对 SPEC 的影响
- AI 推荐模块明确写出输入、行为、输出和降级策略
- 禁止开放式聊天机器人式实现

---

### 3.2 架构方式

#### 决策
采用单仓库前后端分层架构。

#### 原因
- 便于统一管理代码与测试
- 适合课程要求中的模块化推进
- 方便使用 worktree 和 subagent 协作

#### 对 SPEC 的影响
- 架构不再模糊为“前后端分离或轻后端”
- 统一为可控的单仓库分层方案

---

### 3.3 食物库与搜索

#### 决策
- 预置食物和自定义食物共享同一搜索接口
- 搜索结果默认展示中不强制区分来源
- 数据层保留 `sourceType`，支持重名与审计

#### 原因
- 用户体验需要简单直接
- 后端管理需要保留来源信息
- 重名数据在健康类应用里很常见，必须允许存在多个来源

#### 对 SPEC 的影响
- 食物数据模型增加来源字段
- 搜索 API 统一入口
- 任务计划中将搜索与自定义创建拆成两个任务

---

### 3.4 自定义食物归属与数量限制

#### 决策
- 自定义食物只存在对应用户的数据里
- 不与其他用户共享
- 需要限制自定义食物数量

#### 原因
- 避免数据污染和跨用户混淆
- 支持用户私有化管理
- 避免滥用和无上限增长

#### 对 SPEC 的影响
- 所有相关表都需要 `userId`
- API 与仓储层必须按用户隔离
- 需要在验收标准里体现多用户边界

---

### 3.5 趋势视图

#### 决策
趋势视图采用“简单列表 + 最近变化值”，不做复杂图表。

#### 原因
- 与项目规模相符
- 更容易实现和测试
- 更贴近“轻量可用”的产品定位

#### 对 SPEC 的影响
- 体重 / 围度模块不依赖高级可视化组件
- 前端任务可以先保证可读性，再考虑体验增强

---

### 3.6 登录与用户隔离

#### 决策
需要登录与用户隔离。

#### 原因
- 饮食、体重、围度、计划、推荐都属于明显的个人数据
- 课程项目需要体现真实产品数据边界

#### 对 SPEC 的影响
- User 实体成为核心实体之一
- 所有记录表都需要 `userId`
- 前后端任务必须考虑授权/数据隔离

---

### 3.7 容器化策略

#### 决策
单容器优先，只有确实拆前后端服务时才使用 `docker-compose`。

#### 原因
- 控制部署复杂度
- 避免为了形式而增加不必要的运维负担
- 更适合课程项目的交付节奏

#### 对 SPEC 的影响
- 容器化部分以单容器为基线
- `docker-compose` 仅作为扩展选项

## 4. 冷启动验证：使用 workbuddy 推进 T3

### 4.1 验证设置

#### 说明
为验证 `SPEC.md` 与 `PLAN.md` 是否足够清晰，我使用了另一个不同类型的智能体 `workbuddy` 进行冷启动验证。

#### 验证方式
- 不向 `workbuddy` 提供我与主 agent 的聊天历史
- 仅提供 `SPEC.md` 与 `PLAN.md`
- 让它自主推进 `T3. 食物库与搜索 API`

#### 目标
检查陌生智能体是否能够仅凭文档顺利推进任务，而不需要额外隐含上下文。

---

### 4.2 `workbuddy` 提出的关键问题

#### 依赖顺序
`workbuddy` 询问：
- 是否需要先做 `T1 → T2 → T3`
- 还是只先做后端 API 部分

这说明虽然 `PLAN.md` 写了任务依赖，但对陌生执行者来说，推进策略还需要更明确。

#### 数据库选型
`workbuddy` 询问：
- 使用 MySQL 还是 SQLite

这说明 `SPEC.md` 里“可选”的数据库写法，对冷启动实现者而言仍然不够定。

#### 预置食物数据量
`workbuddy` 询问：
- 预置食物放 20–30 种、50 种，还是 100 种

这说明“简化版即可”这个表述对于实现层面来说太抽象，需要明确数量级。

#### 自定义食物上限
`workbuddy` 询问：
- 单个用户最多可以创建多少个自定义食物

这说明“自定义食物数量需要受限”还不足以直接指导实现。

---

### 4.3 我的正式回答

#### 推进策略
我确认采用 `T1 → T2 → T3` 依次实现，而不是只做后端 API。

#### 数据库
我确认使用 MySQL。

#### 预置食物数量
我确认预置 100 种常见食物。

#### 自定义食物上限
我确认单个用户最多创建 50 个自定义食物。

---

### 4.4 冷启动验证暴露出的 SPEC 问题

#### 问题 1：执行顺序不够硬
虽然 `PLAN.md` 已经写了依赖关系，但没有在 T3 目标中明确“必须先完成 T1 和 T2”。

#### 问题 2：数据库选型没有拍板
`SPEC.md` 先前写的是 MySQL 或 SQLite，导致陌生智能体需要再次追问。

#### 问题 3：预置数据规模不够具体
“简化版即可”无法直接转化为实现规模。

#### 问题 4：自定义食物上限不够具体
“需要受限”无法指导开发者实现阈值逻辑。

---

### 4.5 冷启动验证后的文档修订

#### `SPEC.md`
- 数据库选型明确为 **MySQL**
- 自定义食物上限写死为 **每用户 50 个**
- 数据约束中补充了自定义食物上限

#### `PLAN.md`
- T3 目标中补充了冷启动验证确认的顺序、数据库和数据规模
- T3 的实现要点中补充了预置食物初始规模 100 条以及数据层来源字段要求
- T1/T2/T3 的依赖关系在总览中更明确地指向 `T1 → T2 → T3`

## 5. 计划阶段的关键整理

在 `writing-plans` 阶段，需求被拆分为 13 个任务，以保证每个任务都能由一个 subagent 在一次会话内完成。

### 5.1 任务拆分原则
- 先底座，后功能
- 先数据层，后业务层
- 先纯逻辑，后 UI
- 先核心链路，后补充体验
- 所有任务都需要先写失败测试

### 5.2 任务间依赖关系的确认
- `T1` 提供基础工程脚手架
- `T2` 提供 domain、auth 与 repository 基线
- `T3/T4/T7/T9` 可部分并行展开，但冷启动验证已确认要按 `T1 → T2 → T3` 启动
- `T5/T10` 必须建立在统计和其他业务数据之上
- `T11` 依赖多个业务模块完成后做前端整合
- `T12` 做完整联调与验收补强
- `T13` 放在最后做容器化、CI 和文档收尾

### 5.3 为什么这样拆
这种拆法是为了满足 Superpowers 的子代理工作方式：
- 每个任务足够小
- 每个任务可以单独测试
- 每个任务都能在一个会话中推进到完成
- 失败时容易定位问题

## 6. 仍然保留的开放问题

虽然本轮已经把规格基本定清楚，但仍然有少量实现细节需要后续在执行阶段进一步确认：

1. 微信小程序最终运行形态如何与当前单仓库实现对接
2. AI 推荐调用哪一种 LLM 接口或模拟方式
3. 登录鉴权的具体实现方式
4. 统计和推荐在前后端之间的最终分工

这些问题不会影响当前 SPEC 和 PLAN 的主线，但会在实现阶段影响具体实现细节。

## 7. 过程反思

这次 brainstorming 与冷启动验证的最大价值，不在于“确定了做什么”，而在于“明确了不做什么”和“每个功能为什么存在”。

几个最关键的收获是：

- 如果不先明确目标用户，健康类产品很容易泛化成没有主线的功能拼盘
- 如果不先明确 AI 的边界，推荐模块会滑向不可控聊天机器人
- 如果不先明确是否需要登录与隔离，后续数据模型和 API 会反复返工
- 如果不先明确小程序布局感与最终上线目标，前端任务的设计会失焦
- 如果不做冷启动验证，很多“看似清楚”的说明在陌生智能体面前仍然不够精确

### Superpowers brainstorming 做得好的地方
- 能主动追问关键边界，而不是直接把模糊需求翻译成代码
- 能逼迫我把“想法”转成“可执行约束”
- 能帮助我把功能、架构和产品形态同步收敛

### 让我不满的地方
- 某些问题仍然需要我二次追问后才真正落定，说明它对项目边界的收敛还不够“硬”
- 如果我不主动把数字、上限、数据规模写死，陌生智能体仍然会卡住
- 在实现层面，它更擅长提出问题，但最终的具体阈值和边界仍需要人类拍板

因此，这份 SPEC 的价值主要在于：它不仅列出了功能，还明确了产品形态、实现原则和边界约束，并且已经经过一次陌生智能体的冷启动验证，能够作为后续实现的稳定基线。
